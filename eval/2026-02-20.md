## 4. Security (★★★★☆)

### Security Analysis

#### 1. Pickle Deserialization (LOW SEVERITY with proper Django settings)
**Location**: `django_glue/proxies/queryset/proxy.py:23,31`
```python
self.encoded_query = base64.b64encode(pickle.dumps(target.query)).decode()
query = pickle.loads(base64.b64decode(encoded_query))
```

**Risk Analysis**: Pickle deserialization can lead to arbitrary code execution if an attacker can control the data being unpickled. However, this data is stored in Django sessions, and the actual risk depends on your session configuration.

**Your risk is LOW if you have these Django settings (defaults since Django 1.6+):**
```python
# These are Django defaults - you're protected if you haven't changed them:
SESSION_ENGINE = 'django.contrib.sessions.backends.db'  # Server-side sessions
SESSION_SERIALIZER = 'django.contrib.sessions.serializers.JSONSerializer'  # JSON, not pickle
```

**Your risk is HIGHER if:**
- Using `SESSION_ENGINE = 'django.contrib.sessions.backends.signed_cookies'` (cookie-based sessions)
- Using `SESSION_SERIALIZER = 'django.contrib.sessions.serializers.PickleSerializer'`
- Your `SECRET_KEY` is compromised
- Attackers have direct write access to your session backend (database/cache)

**Recommendation**: For defense-in-depth, consider replacing pickle with JSON-based query serialization in a future refactor. For now, ensure your Django settings use server-side sessions with JSONSerializer.

#### 2. CSRF Protection (NOT AN ISSUE - Already Protected)
**Location**: `django_glue/views.py:10-11`

**Assessment**: Initially flagged as missing `@csrf_protect` decorator, but further analysis shows CSRF protection is already working correctly through two mechanisms:

1. **Django middleware** (required in settings):
   ```python
   MIDDLEWARE = [
       # ...
       'django.middleware.csrf.CsrfViewMiddleware',  # Must be present
       # ...
   ]
   ```

2. **JavaScript client sends CSRF token** (`client_js/src/http.js:34-36`):
   ```javascript
   if (requestOptions.csrfProtected) {
       options.headers['X-CSRFToken'] = getHttpCookie('csrftoken')
   }
   ```

**Required Django Settings**:
```python
MIDDLEWARE = [
    # ...
    'django.middleware.csrf.CsrfViewMiddleware',  # Required for CSRF protection
    # ...
]
```

**Status**: No fix needed. The middleware approach is the standard Django pattern.

#### 3. XSS via `|safe` Filter (FIXED)
**Location**: `templates/django_glue/django_glue.html`

**Previous Issue**: The `|safe` filter disabled Django's auto-escaping, potentially allowing XSS if context data contained user input.

**Fix Applied**: Replaced `|safe` with Django's `json_script` template tag:
```html
{{ DJANGO_GLUE_SESSION_PROXY_REGISTRY|json_script:"glue-proxy-registry" }}
{{ DJANGO_GLUE_PROXIES_CONTEXT_DATA|json_script:"glue-context-data" }}
```

The `json_script` filter properly escapes HTML characters (e.g., `<script>` becomes `\u003Cscript\u003E`), preventing XSS attacks.

**Status**: RESOLVED

#### 4. Unrestricted QuerySet Filter (FIXED)
**Location**: `django_glue/proxies/queryset/proxy.py`

**Previous Issue**: The `filter()` method accepted any Django ORM filter syntax, allowing queries on excluded fields.

**Fix Applied**: Added `_validate_filter_keys()` method that validates all filter keys against `_included_fields`:
```python
def _validate_filter_keys(self, payload: dict) -> None:
    for key in payload.keys():
        base_field = key.split('__')[0]
        if base_field not in self._included_fields:
            raise ValueError(f"Cannot filter on field '{base_field}'...")

@action(access=GlueAccess.VIEW)
def filter(self, payload: dict):
    self._validate_filter_keys(payload)
    return self._queryset_to_list(self.target.filter(**payload))
```

Now attempts to filter on excluded fields raise a `ValueError`:
```javascript
// This will now raise an error:
filter({'password__startswith': 'abc'})
// ValueError: Cannot filter on field 'password'. Allowed fields: ['id', 'title', ...]
```

**Status**: RESOLVED

#### 5. No Field Type Validation on Save (LOW SEVERITY)
**Location**: `django_glue/proxies/model/proxy.py:85-92`
```python
for field_name, field_data in payload.items():
    if field_name in self._included_fields:
        setattr(instance, field_name, field_data)  # No type validation
```

**Risk**: Could set DateField to non-date value, ForeignKey to invalid ID, etc.

**Mitigating Factors**:
- Django's ORM will raise exceptions for invalid types on `save()`
- Database constraints provide a safety net
- Pydantic validates request structure (though not field types)

**Recommendation**: Consider adding type coercion/validation before `setattr` for better error messages, but this is not a critical security issue.

### Required Django Settings for Security

To ensure Django Glue operates securely, verify these settings in your Django project:

```python
# settings.py

# Session Security (these are Django defaults, but verify they haven't been changed)
SESSION_ENGINE = 'django.contrib.sessions.backends.db'  # Or 'cache', 'file' - NOT 'signed_cookies'
SESSION_SERIALIZER = 'django.contrib.sessions.serializers.JSONSerializer'  # NOT PickleSerializer
SESSION_COOKIE_HTTPONLY = True  # Prevents JavaScript access to session cookie
SESSION_COOKIE_SECURE = True  # Required for production (HTTPS only)
SESSION_COOKIE_SAMESITE = 'Lax'  # Prevents CSRF via cross-site requests

# CSRF Protection (required)
MIDDLEWARE = [
    # ...
    'django.middleware.csrf.CsrfViewMiddleware',  # Must be present
    # ...
]

# Keep SECRET_KEY secure
SECRET_KEY = 'your-secret-key-here'  # Never commit to version control
```

## Conclusion

Django Glue demonstrates excellent architectural decisions and API design. The proxy pattern implementation is clean, the integration is non-invasive, and the developer experience is intuitive.

**Security Assessment**: With proper Django settings (server-side sessions, JSONSerializer, CsrfViewMiddleware), the library's security posture is reasonable. The remaining concerns are:
- XSS via `|safe` filter (should fix)
- Unrestricted QuerySet filter (should fix)

**Production Readiness**: The library can be used in production with the following caveats:
1. Ensure Django session settings use defaults (server-side sessions, JSONSerializer)
2. Be aware that `filter()` accepts arbitrary ORM lookups until restricted
3. Test coverage should be added before relying on the library for critical applications

With the recommended fixes implemented, Django Glue would be a solid choice for seamlessly binding Django models to frontend JavaScript.
