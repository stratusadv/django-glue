## Changes Implemented

### 1. Payload Validation for Actions

Created a validation system that validates payload field values against Django model field types before applying changes.

**Files Created:**
- `django_glue/proxies/fields/` - New module for field handling
  - `__init__.py` - Exports `GlueProxyFieldsMixin`
  - `validators.py` - Individual validator functions and `FIELD_TYPE_VALIDATORS` mapping
  - `mixin.py` - `GlueProxyFieldsMixin` with `_validate_payload()` method

**Files Modified:**
- `django_glue/exceptions.py` - Added `GluePayloadValidationError`
- `django_glue/proxies/model/proxy.py` - Integrated validation in `save()`
- `django_glue/proxies/queryset/proxy.py` - Updated import path

**Validator Coverage:**
- Integer fields (AutoField, IntegerField, BigIntegerField, etc.)
- String fields (CharField, TextField, EmailField, URLField, etc.)
- Boolean fields (BooleanField, NullBooleanField)
- Date/time fields (DateField, DateTimeField, TimeField)
- Numeric fields (DecimalField, FloatField)
- Relation fields (ForeignKey, OneToOneField)
- JSONField

**Architecture Decision:**
Validation was integrated into `GlueProxyFieldsMixin` rather than a standalone module because:
1. Validation depends on `_included_fields` which is a property of the mixin
2. Co-locates related functionality
3. Simpler API: `self._validate_payload(payload)` vs `validate_payload(payload, self._included_fields)`

- `get()` - Returns field definitions and current values
- `validate(payload)` - Validates form data, returns errors
- `submit(payload)` - Validates and saves (for ModelForm)

**Architecture Decision - Form Class Path vs Pickle:**
Used importable class path (`"myapp.forms.MyForm"`) instead of pickle for session storage because:
1. Forms are stateless factories, unlike QuerySets which contain query state
2. Django forms don't pickle cleanly (widgets, validators, bound data)
3. Reduces pickle usage (already flagged as a security concern)

Scaffolded Jest for JavaScript unit testing with Babel for ES module transpilation.

**Files Created:**

- `jest.config.js` - Jest configuration
- `client_js/tests/config.test.js` - Initial test verifying setup works

**Files Modified:**
- `package.json` - Added test scripts and Babel dependencies
- `CLAUDE.md` - Documented testing setup and Babel rationale

**Why Babel for Jest:**
Jest runs in Node.js which defaults to CommonJS, but our source uses ES modules. We evaluated esbuild-based alternatives:
- `esbuild-jest` - Abandoned (last updated 5 years ago)
- `@jgoz/jest-esbuild` - Niche, ~2 months old
- `esbuild-jest2` - Fork with modest adoption (~4,500 weekly downloads)

Babel + Jest is the mainstream, battle-tested approach with millions of weekly downloads and broad community support. For a library others will depend on, stability trumps marginal speed gains.

### 11. Fixed GlueFormProxy JS Initialization Bug

The form proxy was failing with "Cannot convert undefined or null to object" because `postInit()` was called from the base class constructor before subclass constructor could set `this.fields`.

**Root Cause:**
```javascript
constructor({...}) {
    super({...});  // <-- calls postInit() here!
    this.fields = contextData.fields;  // <-- hasn't run yet
}
```

**Fix:**
Moved field initialization into `postInit()` using `this.contextData` which is available:

```javascript
postInit() {
    this.fields = this.contextData.fields;
    this.values = {...this.contextData.initial};
    // ...
}
```

### 12. Improved JS Proxy API Ergonomics

Made form proxy API match model proxy patterns for consistency.

**Property Accessors:**
- Before: `taskForm.values.title`
- After: `taskForm.title`

**Action Methods - Wrapped Approach:**
Changed both `GlueModelProxy` and `GlueFormProxy` to use wrapped action methods instead of auto-generated ones with `setActionPayload()`.

Before (setActionPayload pattern):
```javascript
set(value) {
    this.values[fieldName] = value;
    this.setActionPayload('save', this.values);
}
// save() auto-generated by base class
```

After (wrapped action pattern):
```javascript
async save() {
    const result = await this.processAction('save', this.values);
    this.values = result;
    return result;
}

async submit() {
    const result = await this.processAction('submit', this.values);
    this.errors = result.errors || {};
    return result;
}
```

**Collision Prevention:**
Updated `BaseGlueProxy.defineActionsAsProperties()` to skip methods already defined by subclasses:

```javascript
defineActionsAsProperties() {
    Object.keys(this.actions).forEach(actionName => {
        if (typeof this[actionName] === 'function') {
            return;  // Skip - subclass already defined this
        }
        this[actionName] = (payload) => this.processAction(actionName, payload)
    });
}
```

## Architecture Decisions

### Wrapped Actions vs setActionPayload

Chose wrapped actions because:
1. Forms have a "fill everything then submit" flow, not incremental updates
2. Allows handling response (update errors, update values) in one place
3. More explicit - you can see exactly what `submit()` does
4. Consistent API across model and form proxies

### Collision Prevention Strategy

Check `typeof this[actionName] === 'function'` in base class works because:
- Subclass prototype methods exist before constructor runs
- ES6 class methods are defined on the prototype
- `this[methodName]` resolves prototype chain correctly

**New TaskSeeder Implementation:**
```python
from faker import Faker
from test_project.task.models import Task

class TaskSeeder:
    def __init__(self):
        self.fake = Faker()

    def create_task(self) -> Task:
        return Task.objects.create(
            title=self.fake.sentence(nb_words=4)[:50],
            description=self.fake.paragraph(nb_sentences=2),
            done=self.fake.boolean(),
            order=self.fake.random_int(min=1, max=1000),
        )

    @classmethod
    def seed_database(cls, count: int = 10) -> list[Task]:
        seeder = cls()
        return [seeder.create_task() for _ in range(count)]
```

**Why jest-environment-jsdom:**
The JavaScript client uses browser APIs (`document.cookie` for CSRF tokens, `window.location.reload()` for session expiry, `fetch` for HTTP requests). Jest's default Node environment doesn't provide these globals. `jest-environment-jsdom` provides a simulated browser environment via JSDOM, allowing tests to access `document`, `window`, and other browser APIs without manual mocking.
