## Changes Implemented

### 1. Payload Validation for Actions

Created a validation system that validates payload field values against Django model field types before applying changes.

**Files Created:**
- `django_glue/proxies/fields/` - New module for field handling
  - `__init__.py` - Exports `GlueProxyFieldsMixin`
  - `validators.py` - Individual validator functions and `FIELD_TYPE_VALIDATORS` mapping
  - `mixin.py` - `GlueProxyFieldsMixin` with `_validate_payload()` method

**Files Modified:**
- `django_glue/exceptions.py` - Added `GluePayloadValidationError`
- `django_glue/proxies/model/proxy.py` - Integrated validation in `save()`
- `django_glue/proxies/queryset/proxy.py` - Updated import path

**Validator Coverage:**
- Integer fields (AutoField, IntegerField, BigIntegerField, etc.)
- String fields (CharField, TextField, EmailField, URLField, etc.)
- Boolean fields (BooleanField, NullBooleanField)
- Date/time fields (DateField, DateTimeField, TimeField)
- Numeric fields (DecimalField, FloatField)
- Relation fields (ForeignKey, OneToOneField)
- JSONField

**Architecture Decision:**
Validation was integrated into `GlueProxyFieldsMixin` rather than a standalone module because:
1. Validation depends on `_included_fields` which is a property of the mixin
2. Co-locates related functionality
3. Simpler API: `self._validate_payload(payload)` vs `validate_payload(payload, self._included_fields)`

### 2. Django Form Proxy

Created `GlueFormProxy` to enable form binding between Django and JavaScript.

**Files Created:**
- `django_glue/proxies/form/__init__.py`
- `django_glue/proxies/form/proxy.py` - `GlueFormProxy` implementation
- `client_js/src/proxies/form.js` - JavaScript form proxy
- `test_project/task/forms.py` - Test forms (TaskForm, ContactForm)
- `tests/proxies/form/` - Test suite (38 tests)

**Files Modified:**
- `django_glue/proxies/__init__.py` - Export `GlueFormProxy`
- `django_glue/maps.py` - Added `BaseForm` mapping
- `django_glue/shortcuts.py` - Added `Glue.form()` method
- `client_js/src/proxies/index.js` - Export form proxy

**GlueFormProxy Actions:**
- `get()` - Returns field definitions and current values
- `validate(payload)` - Validates form data, returns errors
- `submit(payload)` - Validates and saves (for ModelForm)

**Architecture Decision - Form Class Path vs Pickle:**
Used importable class path (`"myapp.forms.MyForm"`) instead of pickle for session storage because:
1. Forms are stateless factories, unlike QuerySets which contain query state
2. Django forms don't pickle cleanly (widgets, validators, bound data)
3. Reduces pickle usage (already flagged as a security concern)

### 3. Jest Testing Infrastructure

Scaffolded Jest for JavaScript unit testing with Babel for ES module transpilation.

**Files Created:**
- `babel.config.js` - Babel configuration for Jest
- `jest.config.js` - Jest configuration
- `client_js/tests/config.test.js` - Initial test verifying setup works

**Files Modified:**
- `package.json` - Added test scripts and Babel dependencies
- `CLAUDE.md` - Documented testing setup and Babel rationale

**Why Babel for Jest:**
Jest runs in Node.js which defaults to CommonJS, but our source uses ES modules. We evaluated esbuild-based alternatives:
- `esbuild-jest` - Abandoned (last updated 5 years ago)
- `@jgoz/jest-esbuild` - Niche, ~2 months old
- `esbuild-jest2` - Fork with modest adoption (~4,500 weekly downloads)

Babel + Jest is the mainstream, battle-tested approach with millions of weekly downloads and broad community support. For a library others will depend on, stability trumps marginal speed gains.
